# Map Regex Workspace Redesign

## Goal

Перекомпоновать вкладку «Модификаторы карт» так, чтобы пользователь сначала задавал условия самой карты и наград, затем ограничения формата, после чего выбирал модификаторы. Интерфейс должен явно показывать логику каждого условия и сохранять точную семантику скомпилированного regex.

За основу принят визуальный вариант A «Сценарий по смысловым блокам».

## Scope

Входит в работу:

- новая компоновка только вкладки `#/regex/maps`;
- переименование и перегруппировка существующих настроек;
- более понятные взаимоисключающие состояния контролов;
- точная генерация regex с лимитом одного игрового запроса в 250 символов;
- сохранение и безопасная миграция существующего профиля;
- RU/EN-паритет, accessibility, адаптивность и regression-тесты.

Не входит:

- изменение исходного каталога модификаторов карт;
- пересмотр сокращённых regex-токенов;
- изменение других генераторов;
- добавление новых игровых фильтров, которых нет в текущем компиляторе.

## Information Architecture

Порядок блоков сверху вниз:

1. Заголовок вкладки и возврат в каталог.
2. Закреплённый результат regex.
3. Две равноправные смысловые панели:
   - «Пороги самой карты»;
   - «Награды качества».
4. Компактный блок «Формат карты» с редкостью, осквернением и опознанием.
5. Поиск, счётчик активных условий и сброс.
6. Сетка карточек модификаторов.

На узких экранах панели порогов складываются в одну колонку, а результат остаётся первым блоком. Боковая липкая панель не используется.

## Regex Result

Результат расположен сверху и содержит:

- моноширинный текст запроса;
- счётчик `текущая длина / 250`;
- кнопку копирования;
- diagnostics без скрытого обрезания выражения.

Если точное выражение помещается в один запрос, показывается один блок. Если оно превышает 250 символов, разделение разрешается только тогда, когда компилятор способен доказать семантическую эквивалентность двух проходов и вернуть способ их композиции:

- `union` — выполнить A и B отдельно и объединить найденные предметы;
- `intersection` — выполнить B только на подмножестве, прошедшем A.

Для безопасного разложения показываются два самостоятельных блока «Проход A» и «Проход B» с отдельными кнопками копирования и конкретной инструкцией по композиции результатов. Смешанное выражение, которое нельзя представить двумя проходами без ложных совпадений или пропусков, не разделяется.

Если безопасное разделение невозможно, интерфейс показывает блокирующую диагностику и не выдаёт заведомо усечённый regex.

Поле «Своё выражение» остаётся доступным в компактном раскрываемом блоке «Дополнительное условие» рядом с результатом. Оно не занимает постоянное место, пока пользователь его не открыл.

## Map Thresholds

Панель «Пороги самой карты» содержит четыре числовых поля:

- минимальное количество предметов;
- минимальный размер групп;
- минимальная редкость предметов;
- минимальный шанс выпадения карт.

Пустое поле не участвует в regex. Некорректное или нулевое значение нормализуется текущими правилами компилятора и не создаёт ложного условия.

Три существующих переключателя округления убираются из основного интерфейса. Компилятор по умолчанию строит точные пороги. Это соответствует текущему лимиту PoE 1 в 250 символов, введённому в 3.26.

Если точное выражение не помещается, интерфейс сначала использует безопасное разделение A/B. Вторичная оптимизация «Уместить в один запрос» может быть предложена только тогда, когда компилятор способен явно сообщить пользователю об изменении порогов; она не включается автоматически и не входит в первый implementation pass.

## Quality Rewards

Панель «Награды качества» содержит шесть числовых полей:

- количество на карте;
- количество валюты;
- гадальные карты;
- редкость награды;
- размер групп в награде;
- скарабеи в награде.

Внутри панели находится явный двухпозиционный переключатель:

- «Любое (ИЛИ)» — значение по умолчанию и эквивалент текущего `anyQuality: true`;
- «Все (И)» — эквивалент текущего `anyQuality: false`.

Пустые поля не участвуют ни в ИЛИ, ни в И. Под заголовком панели динамически отображается одно из пояснений:

- «Карта должна выполнить любое из заполненных условий»;
- «Карта должна выполнить все заполненные условия».

Переключатель больше не показывается среди глобальных галок, поскольку он относится только к этим шести полям.

## Map Format

Существующие пары «Фильтровать» и «Включать» заменяются взаимоисключающими состояниями.

Осквернение:

- «Любое»;
- «Только осквернённые»;
- «Исключить осквернённые».

Опознание:

- «Любое»;
- «Только неопознанные»;
- «Исключить неопознанные».

Редкость карты остаётся множественным выбором из «Обычная», «Волшебная», «Редкая». Когда выбраны все три, условие редкости не добавляется. Нулевой выбор считается недопустимым UI-состоянием: последнюю активную редкость нельзя выключить.

Новые контролы являются представлением существующих полей `rarity`, `corrupted` и `unidentified`; формат хранимого профиля менять необязательно.

## Modifier Cards

Каждая карточка имеет одно из трёх взаимоисключающих состояний:

- «Не учитывать»;
- «Исключить»;
- «Требовать».

Выбор «Исключить» добавляет ID только в `badIds`, выбор «Требовать» — только в `goodIds`, нейтральное состояние удаляет ID из обоих массивов. Невозможное состояние одновременного присутствия ID в обоих массивах устраняется на уровне UI и нормализации профиля.

Визуальные признаки:

- исключённая карточка получает красный акцент;
- требуемая — зелёный;
- мод Кошмара — отдельный фиолетовый бейдж независимо от выбранного состояния;
- нейтральная карточка не получает смысловой цвет.

Над сеткой располагаются:

- поиск;
- «Показывать моды Кошмара»;
- переключатель логики требуемых модов «Все / Любой»;
- счётчик реально активных, а не скрытых выбранных модов;
- сброс.

Скрытые моды Кошмара не отображаются, не компилируются и не учитываются в активном счётчике, но сохранённый выбор не уничтожается и восстанавливается после повторного включения отображения.

## State and Data Flow

`MapRegexSettings` остаётся единственным источником состояния вкладки. UI-контролы преобразуют пользовательские действия в существующие поля настроек, после чего `compileMapRegex` формирует результат. Профиль сохраняется после каждого изменения через существующий storage pipeline.

При загрузке старого профиля:

- `anyQuality` напрямую выбирает ИЛИ или И;
- пары `enabled/include` преобразуются в одно из трёх визуальных состояний;
- конфликтующие `badIds`/`goodIds`, если они существуют, разрешаются детерминированно нормализатором;
- отсутствующие поля получают текущие безопасные значения по умолчанию.

## Accessibility and Responsive Behaviour

- Каждая группа имеет `fieldset` и видимый `legend` либо эквивалентную доступную группу.
- Сегментированные контролы доступны с клавиатуры и сообщают активное состояние через native radio semantics или `aria-pressed`.
- Цвет не является единственным признаком состояния карточки: остаётся текст выбранного действия.
- Фокус видим на всех интерактивных элементах.
- На мобильном экране панели, формат и карточки переходят в одну колонку без горизонтального скролла.

## Testing

Unit/regression coverage:

- ИЛИ и И для заполненных quality-порогов;
- игнорирование пустых quality-полей;
- все три состояния осквернения и опознания;
- минимум одна выбранная редкость;
- взаимоисключение `badIds` и `goodIds`;
- скрытие модов Кошмара в списке, компиляторе и счётчике;
- точный результат до 250 символов;
- семантически доказанный A/B split с режимом `union` или `intersection`;
- blocking diagnostics для смешанного или слишком длинного выражения, которое нельзя разложить без изменения результата;
- миграция и round-trip старого профиля;
- точный RU/EN message parity.

Browser smoke coverage:

- desktop и mobile layout без overflow;
- клавиатурное переключение сегментов и карточек;
- копирование одного запроса и проходов A/B;
- восстановление настроек после reload;
- отсутствие console errors.

Release gate остаётся `npm run verify`, после чего GitHub Pages проверяется на публичном URL с cache-buster.

## Acceptance Criteria

- Качество наград визуально и логически отделено от порогов самой карты.
- Пользователь без знания внутренней модели понимает разницу ИЛИ и И.
- В интерфейсе нет пар неоднозначных галок «Фильтровать/Включать».
- Карточка модификатора не может одновременно быть исключённой и требуемой.
- Точный regex никогда молча не округляется и не обрезается.
- Один игровой запрос не превышает 250 символов; длинные результаты представлены как A/B только с доказанной композицией `union`/`intersection`, иначе блокированы диагностикой.
- Существующие профили открываются без потери пользовательского выбора.
- Вкладка сохраняет текущую палитру PoE Tools и компоновку варианта A.
